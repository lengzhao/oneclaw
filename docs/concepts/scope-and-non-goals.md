# 范围与非目标

本文固定 `oneclaw` 当前阶段的核心范围边界，供 `concepts/`、`architecture/` 与后续 ADR 引用。

如果能力方向变化，应先改本文件，再改相关 ADR 与参考文档。

## 当前明确承诺

### 1. 主语言与运行时形态

- 以 **Go** 为主语言实现运行时内核与 CLI/服务。
- 日志统一使用标准库 `log/slog`。
- 仓库约定：**不使用** `internal/` 目录。

### 2. 工具扩展路径

- 工具层优先支持 **MCP** 与静态 `ToolRegistry` 并列。
- Host 负责连接聚合、生命周期与健康检查。
- 首版重点是跑通 `Loop + Session + Bus + MCP 工具发现/执行`。

### 3. 自我进化的默认方向

- 默认进化优先落在 **记忆、知识、SOP、Skills** 这些外部化载体。
- 默认目标是低门槛使用，而不是先暴露复杂多 Agent 配置。
- 高风险进化必须可审计、可回滚、可限制写入范围。
- 版本管理与回滚默认直接依赖 git，而不是另造一套独立版本系统。

## 当前不承诺

### 1. OpenClaw Gateway 协议兼容

- 首版不追求与 OpenClaw Gateway WebSocket 控制面字节级或消息级兼容。
- 若未来需要兼容，应通过独立适配层评估，而不是反向约束内核设计。

### 2. 重度产品控制面

- 不把完整 Host+UI 产品、控制台、移动节点、多通道面板列为内核同里程碑交付物。
- 这些能力可以由宿主、子项目或后续产品层承担。

### 3. 默认多 Agent 门槛

- 不要求使用者先理解复杂路由或 profile 编排，才能获得记忆与知识沉淀能力。
- 多 Agent 是渐进升级路径，而不是默认必选项。

## 范围判断原则

遇到新需求时，优先问这三个问题：

1. 这是内核职责，还是宿主/产品职责？
2. 这会不会把默认体验从“低门槛”推向“重配置”？
3. 这是不是应该通过外部化载体与治理策略解决，而不是往 `agent.Loop` 加更多隐式行为？
