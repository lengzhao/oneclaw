# oneclaw 自我进化能力验证方案 v0

> 日期：2026-04-05
> 目标：用一个最小、可复现、可量化的实验，验证 oneclaw 是否具备“根据反馈改进后续行为”的能力。

## 1. 验证目标

本次不验证“模型本身变聪明”，而验证 oneclaw 运行时是否能够完成下面这个闭环：

1. 执行任务
2. 暴露失败或低质量行为
3. 从失败中提取可复用规则
4. 将规则写入可持久化载体（memory / rules / agent memory）
5. 在后续相似任务中自动应用该规则
6. 使结果优于第一次尝试

如果以上 6 步成立，就可以认为 oneclaw 至少具备 **Level 1 的自我进化能力：行为修正**。

---

## 2. 本次实验范围

优先验证一个最小能力点：

**能力点：收到失败反馈后，是否会调整工具使用策略，并在下一次相似任务中稳定复用。**

建议选取最容易观测的策略规则：

- 编辑文件前先读文件
- 修改前先查看相关上下文
- 失败后写入 memory
- 后续任务中主动读取并遵守该 memory

本轮建议使用最简单的一条：

### 推荐验证规则

**规则：在编辑文件前，必须先读取目标文件或相关上下文文件。**

这是一个非常适合做 MVP 的点，因为它具备：

- 容易人为构造失败场景
- 容易通过工具调用轨迹验证
- 容易写成明确规则
- 容易在第二次任务中观察是否生效

---

## 3. 实验假设

### H1：可学习
当 oneclaw 在第一次任务中因“未先读文件就修改”而失败后，能够从反馈中提炼出规则。

### H2：可持久化
该规则能够被写入文件化 memory 或 rules，而不是只停留在当前上下文。

### H3：可迁移
在第二个相似任务中，oneclaw 会主动应用该规则，表现出更合理的工具调用顺序。

### H4：可量化改善
第二次任务相较第一次，在正确率、返工次数或工具调用顺序上有可观察提升。

---

## 4. 成功标准

满足以下条件可判定本轮实验成功：

### 必选标准

1. **首次实验存在明确失败或可归因缺陷**
   - 例如：直接写文件，导致改错位置、覆盖错误内容、遗漏上下文

2. **产生明确规则**
   - 例如：`编辑前先 read_file，必要时读取相关依赖文件`

3. **规则被持久化**
   - 写入 `.oneclaw/memory`、agent memory、project memory 或规则文件

4. **第二次相似任务中自动应用规则**
   - 不需要用户再次提醒

5. **第二次结果优于第一次**
   - 成功率更高，或明显减少错误/返工

### 加分标准

- 规则不是只对单个文件有效，而是抽象成通用策略
- 在第三个相似任务中也继续生效
- 规则没有引入明显副作用（例如过度读取无关文件）

## 5. 实验设计

## 场景 A：工具使用顺序纠偏

### 任务类型
给 oneclaw 一个需要修改现有文件的任务。

### 首次任务设计
人为选择一个“如果不先读文件就容易出错”的文件。

示例：
- 修改 README 某一段描述
- 调整配置项名称
- 修改 prompt 模板中的一条规则
- 更新某个 Go 文件中的常量或函数签名

### 期望的首次失败
观察 oneclaw 是否出现以下问题之一：

- 没有先 `read_file`
- 只凭猜测改内容
- 忽略相邻上下文
- 改动与实际文件结构不匹配
- 产生返工

### 反馈语
用户可在首次失败后给出明确反馈，例如：

> 你刚才的问题是修改前没有先看文件上下文，导致改错了。以后编辑前先读取目标文件。

### 学习动作
期望 oneclaw：

- 提炼规则
- 写入 memory / rules
- 在后续任务中使用该规则

### 第二次任务设计
给出另一个相似但不完全相同的编辑任务，例如：

- 第一次改 `README.md`
- 第二次改 `docs/README.md`

或：

- 第一次改某个 prompt 文件
- 第二次改另一个 prompt 文件

### 验证点
检查第二次任务是否：

1. 先 `read_file`
2. 再决定是否修改
3. 最终改动更准确

---

## 6. 度量指标

建议记录以下指标：

### 行为指标

- 是否先读取目标文件：`yes / no`
- 是否读取了必要上下文：`yes / no`
- 是否发生返工：`0/1/2...`
- 工具调用顺序是否合理：`good / bad`

### 结果指标

- 任务是否一次成功：`yes / no`
- 最终输出是否符合要求：`yes / no`
- 错误类型数量：整数

### 进化指标

- 是否生成新规则：`yes / no`
- 是否写入持久化 memory：`yes / no`
- 后续任务是否自动触发该规则：`yes / no`
- 规则是否跨任务迁移：`yes / no`

### 建议的最小量化结论

可用如下方式给出结论：

- 首次任务：先读文件 = 否；任务成功 = 否
- 第二次任务：先读文件 = 是；任务成功 = 是
- 判定：出现可观测行为修正，初步验证自我进化能力成立

## 8. 建议的持久化位置

如果这类规则是项目级通用规则，建议优先写到：

- project memory
- `.oneclaw/memory/MEMORY.md` 的关联 topic
- agent memory（偏执行策略）

如果后续需要更正式的规则系统，可以再引入：

- `.oneclaw/rules/editing.md`
- `AGENT.md`

本轮 MVP 不要求把规则系统做复杂，只要能证明：

**规则被写下来了，而且后续真能用。**

---

## 9. 自我进化能力的分层定义

为了避免“写了一条 memory 就算进化”的误判，建议把能力分三层：

### Level 1：行为修正
- 能根据失败反馈调整下一次行为

### Level 2：规则抽象
- 能把单次失败总结成通用规则

### Level 3：跨任务迁移
- 能在不同但相似任务中应用该规则

本周建议先把 **Level 1 + Level 2 的最小证明** 做出来；如果顺利，再继续验证 Level 3。

## 10. 执行建议

### 今日可直接落地的步骤

1. 选一个容易出错的编辑任务
2. 记录 oneclaw 第一次尝试
3. 明确指出失败原因
4. 让 oneclaw 生成并持久化规则
5. 给第二个相似任务
6. 检查行为是否变化
7. 填写实验记录

### 最小里程碑

如果今天能拿到下面这个结果，就算非常有价值：

- 第一次：失败
- 中间：生成规则并写入 memory
- 第二次：主动先读文件，再成功修改

这就已经足够支撑 README 或发布说明里的一句话：

> oneclaw 已初步验证基于文件化记忆的行为修正闭环，可在失败反馈后调整后续工具使用策略。

## 11. 后续扩展实验

在 MVP 跑通后，可以继续验证下面几类“进化”：

1. **偏好学习**
   - 例如记住用户称呼、回答风格、文档格式偏好

2. **代码库策略学习**
   - 例如在本仓库中优先查看 `docs/` 再改 README

3. **失败规避学习**
   - 例如 shell 命令失败后自动降低风险、缩小搜索范围

4. **维护蒸馏能力**
   - 例如从 daily log 合并出稳定 topic，而不是无限追加噪声

5. **跨 agent 共享规则**
   - 主 Agent 学到的规则，是否可安全传递给子 Agent

---

## 12. 当前建议结论

对于 oneclaw 当前阶段，最值得优先验证的不是“大而全的自主进化”，而是：

**一个有证据的、自洽的、小闭环。**

建议本轮就聚焦于：

- 失败反馈
- 规则提炼
- 文件化持久化
- 后续复用
- 明确改善

只要这个闭环跑通，oneclaw 就已经从“具备 memory 能力”迈向了“具备可验证的自我进化能力”。
