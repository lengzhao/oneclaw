# E2E 用例清单（真实 LLM / Live）

与 **[CASES.md](./CASES.md)**（`openaistub`、默认 CI）分工如下：

| 方式 | 负责范围 |
|------|----------|
| **Stub** | 异常路径、边界条件、鉴权失败、空输入、工具拒绝、确定性编排、审计 JSONL 结构等——**全部在 Stub 侧覆盖即可**。 |
| **真实 LLM** | **自学 / 进化链路是否在真实模型下「真的生效」**：文件型记忆平面是否更新、维护/摘要是否改善后续可用性、与**设计预期**相比有何**差异**。 |

「自学 / 进化」在本项目中的含义见 [`docs/todo.md`](../../docs/todo.md)：**不是微调权重**，而是 **memory 落盘持续更新** + **维护型子任务整理** + **规则/策略可写回**（与 Stub 已测的管道逻辑互补；Live 关心的是 **LLM 生成内容质量与后续轮次行为**）。

**不默认进 CI**（费用、时延、非确定性）。

---

## 1. 环境与门禁

| 项 | 说明 |
|----|------|
| **客户端** | **不要**把 `openai.base_url` 指到 `openaistub`；使用官方或兼容网关（见 `live_llm.config.yaml`）。 |
| **鉴权** | 合并 YAML 中的 `openai.api_key`（见 [`docs/config.md`](../../docs/config.md)）。 |
| **模型** | YAML `model`；**同一套 Live 场景建议固定模型与版本**，便于对比「预期 vs 实际」的时间序列。 |
| **传输** | 若产品同时支持 `stream` / `non_stream`，进化链路（PostTurn、维护）建议在**生产默认传输**下各验收一次。 |
| **隔离** | `t.TempDir()` 为 cwd；`HOME` 与 YAML `paths.memory_base`（测试中可用 `e2eIsolateUserMemory`）指向临时目录，避免污染本机。 |
| **门禁** | `RUN_LIVE_LLM=1` + `testing.Short()` 跳过；避免 CI 无密钥仍跑 Live。 |

## 2. 验收核心：预期、实际与差异

Live 的价值是发现 **Stub 测不到的模型侧问题**（胡编 bullet、忽略 recall、摘要丢关键约束等）。每条用例建议按下面结构记录（可写在 PR、发布说明或独立 `live-runs.md` 中）：

| 字段 | 内容 |
|------|------|
| **设计预期** | 在本轮输入与配置下，**磁盘 / 下一 prompt** 应出现什么（例如：`memory/YYYY-MM-DD.md` 新增与对话一致的要点、规则 `MEMORY.md` 仅含策略性条文、daily log 有一行摘要、维护 prompt 含多日 log 片段等）。 |
| **实际观测** | 本次运行：**文件内容摘录**、**下一轮模型行为**（是否用到记忆中的事实）、**token/轮数**。 |
| **与预期的差异** | 无 / 有：若有，是 **可接受的模型方差** 还是 **产品 bug**（管道未触发、护栏过严、prompt 不清）。 |
| **元数据** | 日期、`model`、网关、合并 YAML / `PushRuntime` 中与 memory、维护、compact、recall 相关的关键项。 |

自动化时：**只对「管道是否触发、文件是否生成、字段是否存在」做强断言**；对 **自然语言质量、是否「聪明地用记忆」** 以人工 + 上表差异记录为主。

## 3. 用例总表（仅自学 / 进化相关）

下列 **LIVE-E** 编号与 [CASES.md](./CASES.md) 中 Stub 用例**对照**，Stub 负责「是否调用、是否落盘、是否去重」；Live 负责「**真模型产出是否可用、下一轮是否变好**」。

| LIVE-E | 摘要 | 对照 Stub（逻辑同源） | 验收焦点（预期 vs 实际） |
|--------|------|------------------------|-------------------------|
| **LIVE-E1** | **Daily log 沉淀**：一轮对话包含明确、可摘要的事实；PostTurn 开启时当日 log 有新增行。 | E2E-50、E2E-51 | **预期**：行内容与对话相关；**差异**：模型摘要跑偏、重复行、或关闭 extract 时无误判为失败。 |
| **LIVE-E2** | **跨轮 / 跨会话可读性**：同一 cwd 下先产生 log 或 topic，再新开会话提问仅依赖「记忆文件」即可回答的问题。 | E2E-30～E2E-33、E2E-20 | **预期**：recall 附件或注入中出现相关片段，模型回答利用该信息；**差异**：附件为空、模型视而不见、或 hallucination。 |
| **LIVE-E3** | **自动维护写回 episodic 日文件**：对话或 log 中含可整理要点；维护模型跑完后 `memory/YYYY-MM-DD.md` 结构与可读性；规则 `MEMORY.md` 保持简短。 | E2E-92、E2E-101、E2E-102 | **预期**：新 bullet 忠实、不重复已有规则/要点（与强去重设计一致）；**差异**：空洞 bullet、重复、与 log 矛盾——记差异表。 |
| **LIVE-E4** | **`oneclaw -maintain-once`（真实模型）**：预置多日 daily log + topic；跑一次维护子命令。 | E2E-96、E2E-113 | **预期**：当日 episodic digest 体现多日 log 与 topic 的合并意图；**差异**：漏天、漏 topic、篇幅失控。 |
| **LIVE-E5** | **语义 compact**：刻意撑满上下文预算，使触发 compact；再继续任务依赖「被摘要」的早期约束。 | E2E-103、E2E-104 | **预期**：长程任务仍满足关键约束（或明确说明摘要边界）；**差异**：摘要丢条件、模型偏离——**最适合记「与预期差异」**。 |
| **LIVE-E6** | **（可选）写回规则 / 策略**：若已实现「行为策略写回 `.oneclaw/rules`」类产品能力，用真实模型验收写回内容与后续注入。 | 待 backlog 落地后补 Stub 对照 | **预期**：写回文件可解析、下一轮遵守；**差异** 记入差异表。 |

**刻意不包含（请用 Stub）**：无 Key、空输入、ctx cancel、越权路径、未知工具名、损坏 transcript、审计 JSONL 纯结构等——见 [CASES.md](./CASES.md) 对应章节。

## 4. 推荐执行顺序

1. **LIVE-E1**：先确认「对话 → 磁盘 log」在真模型下摘要可用。  
2. **LIVE-E2**：再验证「磁盘/召回 → 下一轮回答」是否连贯。  
3. **LIVE-E3 / LIVE-E4**：维护管道与 CLI（成本最高，放后）。  
4. **LIVE-E5**：长上下文与 compact（耗时、需设计固定剧本）。  
5. **LIVE-E6**：随功能交付再开。

---

## 5. 运行方式（建议）

- 默认 **`go test ./test/e2e/...` 仍全部 Stub**。  
- Live 建议 `//go:build live_llm` 或：

```go
if os.Getenv("RUN_LIVE_LLM") != "1" {
    t.Skip("evolution live: set RUN_LIVE_LLM=1")
}
if testing.Short() {
    t.Skip("short")
}
```

- 每条 **LIVE-E** 跑完填写 §2 差异表，便于模型升级后做回归对比。
